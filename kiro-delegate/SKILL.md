# kiro-delegate

ユーザーが「kiroに投げて」「kiroで実装して」など **明示的に** kiro への委譲を指示した場合のみ使用する。
指示がない限り、このスキルを自動発動させてはならない。

## 実行手順

### 0. 事前チェック（必須）

kiro はインタラクティブに停止することがある。タスク実行中にバージョン不足やログイン要求で中断されないよう、開始前にまとめて解決しておく。

#### 0-1. インストール・バージョン確認

1コマンドで確認する：

```bash
VERSION=$(kiro-cli --version 2>/dev/null | grep -oE '[0-9]+\.[0-9]+\.[0-9]+')
if [ -z "$VERSION" ]; then
  echo "NOT_INSTALLED"
else
  MAJOR=$(echo "$VERSION" | cut -d. -f1)
  if [ "$MAJOR" -lt 2 ]; then echo "OLD_VERSION:$VERSION"; else echo "OK:$VERSION"; fi
fi
```

- `NOT_INSTALLED` → 中断してユーザーに伝える：「`kiro-cli` がインストールされていません。インストール後に再実行してください。」
- `OLD_VERSION:X.Y.Z` → 中断してユーザーに伝える：「kiro-cli のバージョンが古いです（現在: X.Y.Z、必要: 2.0.0 以上）。`kiro-cli update` を実行してください。」
- `OK:X.Y.Z` → 次のステップへ

#### 0-2. ログイン確認・自動ログイン

タスク途中でログイン要求が発生すると kiro が停止してしまうため、必ず事前に確認する。

```bash
kiro-cli whoami 2>&1; echo "EXIT:$?"
```

exit code が 0 であればログイン済みなので次のステップへ。非0 の場合はユーザーに「ログインを開始します」と伝えてから実行する：

```bash
kiro-cli login
```

`timeout` は 120000（2分）以上を設定してブラウザ認証の完了を待つ。完了後 `kiro-cli whoami` で成功を確認してからタスクへ進む。

### 1. タスクを自己完結した形に整理する

kiro はインタラクティブに質問できないため、曖昧な指示では誤った判断をしたり途中で止まったりする。
タスク説明は以下を含めること：

- **対象ファイルと場所**: `src/foo.py` の line 42 など、kiro が迷わず作業できる粒度
- **期待する変更の詳細**: 変更前と変更後のコードや文章を明示する（「修正して」だけでは意図が伝わらない）
- **完了条件**: 「品質チェックを実行して通過すること」など kiro が自己検証できる基準

> **Before/After コードの書き方について**:
> kiro は対象コードを見つけるために内部でファイル検索ツールを使う。
> Before 例に `{`, `}`, `.`, `*`, `(`, `)` などの正規表現特殊文字が含まれると、
> 検索パターンとして解釈されツールのバリデーションエラーが発生することがある。
> **ファイルパスと行番号を明示し**、Before/After はあくまで「意図の説明」として添えること。
> （例: "`src/foo.py` の line 42 の `config = {"key": "value"}` 行を修正する"）

> **複数ファイルをまとめて委任する場合**:
> ファイルごとにセクションを分けて記述する（`### ファイル1: ...` / `### ファイル2: ...`）。
> 同じパターンの変更であっても、対象メソッドと行番号をファイルごとに列挙すること。

> **TDD でインポートが削除される場合**:
> 修正によってモジュールから特定シンボルのインポートが消える場合、
> `patch("module.Foo")` を使うテストは `AttributeError` で失敗する。
> この場合は kiro にテストアプローチの変更を許容させる（kiro は自律的に回復できる）か、
> プロンプトに「モジュールが X をインポートしていないことを構造的に検証するテストを書く」と明示しておくと安定する。

> **ブランチ作成を含むタスクのベース指定**:
> kiro が `git checkout -b <branch>` を実行する時点でどのブランチにいるかは制御できない。
> 作業ブランチに未マージのコミットが残っている場合、それらがそのまま新ブランチに取り込まれてしまう。
> タスク説明のブランチ作成手順は必ず **`git checkout <base-branch> && git checkout -b <new-branch>`** の形式で記述すること。
> （例: `git checkout main && git checkout -b fix/my-feature`）

### 2. kiro-cli を実行する

**重要**: kiro はコード変更後にプロジェクトの品質チェック（テスト・型チェック・lint など）を実行するため、
コードの変更規模が小さくても完了まで数分かかることが多い。

#### タイムアウトの目安

タスクの規模に応じて `timeout_ms` を設定する。テスト実行が含まれるため、コード変更が小さくても時間がかかる：

| 規模 | 目安 | timeout_ms |
|---|---|---|
| 小（1〜2ファイル） | 6〜10分 | 600000 |
| 中（3〜5ファイル） | 15〜20分 | 1200000 |
| 大（6ファイル以上・テスト多数） | 25〜30分 | 1800000 |

タイムアウトで途中終了した場合は、完了済み部分をプロンプトに明記して再実行する。

#### 2-a. タスク前の使用量を記録する（Bash）

Monitor を挟むとシェル変数のスコープが切れるため、一時ファイルに保存する。
ファイル名は UUID を使い、同名衝突を防ぐ。

```bash
KIRO_USAGE_FILE="/tmp/kiro-usage-before-$(uuidgen | tr '[:upper:]' '[:lower:]').txt"
echo "/usage" | kiro-cli chat --no-interactive 2>&1 | sed 's/\x1b\[[0-9;?]*[mGHJKlhABCDfr]//g' | grep -v '^$' > "$KIRO_USAGE_FILE"
```

#### 2-b. タスク内容を一時ファイルに書き出す（Bash）

heredoc 内の特殊文字エスケープを避けるため、まずタスク説明をファイルに保存する。
ファイル名は UUID v7 相当のランダム値を使い、同名衝突を防ぐ。

```bash
KIRO_TASK_FILE="/tmp/kiro-task-$(uuidgen | tr '[:upper:]' '[:lower:]').txt"
cat > "$KIRO_TASK_FILE" << 'TASK_EOF'
<タスク説明（複数行）>
TASK_EOF
```

#### 2-c. Monitor でリアルタイム表示しながら kiro を実行する

Bash の代わりに **Monitor ツール** を使って kiro を起動すること。
実行後はタスクファイルを削除する（`; rm -f "$KIRO_TASK_FILE"` を末尾に追加）。

```
Monitor(
  command="kiro-cli chat --no-interactive --trust-all-tools < \"$KIRO_TASK_FILE\" 2>&1 | grep --line-buffered -E 'FAILED|failed|Error:|error:|Traceback|passed in|warnings in|✅|❌|All checks|すべてのチェック|<プロジェクトの品質チェックコマンド>'; rm -f \"$KIRO_TASK_FILE\"",
  description="kiro実行中: <タスク概要>",
  timeout_ms=1200000,   # 規模に応じて調整（上の目安参照）
  persistent=false
)
```

**通知フィルターの方針**:
- 品質チェック結果（`passed in`, `FAILED`, `✅`, `❌`）
- エラー・例外（`Error:`, `Traceback`, `failed`）
- 品質チェックコマンドの実行行（プロジェクトの自動修正・検証コマンド名をフィルターに追加する）
- ルーチンのファイル read/write 成功ログは通知不要のためフィルタで除外する

オプションの意味：
- `--trust-all-tools`: kiro がファイル変更・コマンド実行を確認なしで行う
- `--no-interactive`: kiro はタスク完了後に自動終了する（途中で止まらない）

#### kiro 実行中の対応方針

Monitor から通知が届いても、**エラー・失敗・タイムアウト以外では返信しない**。
kiro のルーチン進捗（ファイル読み込み、検索など）はそのまま垂れ流しにする。
介入が必要な場合のみ：
- `FAILED` / `Error` → 内容を確認して対処を検討。ただし TDD の Red フェーズや `Tool validation failed` など kiro が自律回復中の場合は介入不要（品質チェック通過の通知を待つ）。
- `Monitor timed out` → kiro プロセスが終了しているか確認し、必要なら再実行

### 3. 出力を確認する

**重要**: Monitor が `status: completed` を返すまで次のステップに進まないこと。
kiro は内部で複数のステップを順番に実行する（変更 → クリーンアップ → 品質チェック）。
途中の通知（例: `"N operations processed"`）はまだ作業中の可能性があり、
この時点で差分を確認すると中間状態（重複インポートなど）が見えることがある。

コマンドの返却内容で分岐する：

**A. ファイルに自動保存された場合**（出力が大きすぎてファイルに保存されたと示される場合）

末尾にテスト結果とサマリーが集中しているため、保存されたファイルの末尾付近だけ読めば十分。

**B. インラインで返ってきた場合**

そのままコマンド結果の末尾を読めばよい。

どちらの場合も確認ポイントは同じ：
- 品質チェック（テスト・型チェック・lint）の結果
- kiro のサマリーコメント（何を変更したかの箇条書き）

### 4. 変更内容を検証する

kiro のサマリーは「何をしようとしたか」であり「何が実際に変わったか」ではない。
**`git status --short`** と **`git log <base>..HEAD --oneline`** を並行して確認する。

- **未コミットの変更がある** → 正常（タスクにコミット指示を含めなかった場合）。`git diff` で変更内容を確認する。
- **コミットが存在する** → 要確認。コミットメッセージと差分を精査し、意図しない変更やプッシュがないか確認する。コミット数が想定より多い場合は `git reset --hard <base>` → `git cherry-pick <想定コミット>` で整理する。

差分確認では以下の両面を検証する：

- **意図した変更が行われているか**: 指示した変更箇所がすべて反映されているか
- **意図しない変更がないか**: 無関係なファイルに副作用が出ていないか

kiro が誤った変更をしている場合は、エージェント自身がファイルを直接修正するか、
補足指示を加えて kiro を再実行する。

### 5. ユーザーに報告して後続作業を実施する

- 変更ファイルと変更内容をユーザーに伝える
- 品質チェック結果を報告する
- コミット・プッシュなどの後続作業が必要な場合はエージェント自身が行う（kiro にはさせない）
  kiro はツールを確認なしで使うため、意図せず push などを実行する可能性がある
- タスク完了後に以下のコマンドで使用量を取得し、タスク前後の両方をユーザーに伝える：

```bash
KIRO_USAGE_FILE=$(ls -t /tmp/kiro-usage-before-*.txt 2>/dev/null | head -1)
echo "=== タスク前 ===" && cat "$KIRO_USAGE_FILE"
echo "=== タスク後 ===" && echo "/usage" | kiro-cli chat --no-interactive 2>&1 | sed 's/\x1b\[[0-9;?]*[mGHJKlhABCDfr]//g' | grep -v '^$'
rm -f "$KIRO_USAGE_FILE"
```

### kiro が失敗した場合

出力を確認して原因を特定する：
- **タイムアウト**: `timeout_ms` が小さすぎた → 規模に応じた値に増やし、完了済み部分を明記して再実行
- **品質チェック失敗**: kiro が修正を試みても通過しなかった場合、エージェント自身で残りを修正する
- **タスクが途中で止まった**: 出力末尾を確認して何が問題だったか特定し、補足指示で再実行する
- **MCPサーバーの警告**: 通常は無視してよい（kiro の動作には影響しない）

### kiro の自律回復について

kiro は品質チェック失敗時に自分でアプローチを変更することがある（例: テスト手法の切り替え）。
一定の自律回復が期待できるため、軽微な試行錯誤が出力に現れても問題ない。
ただし最終的な差分は必ず `git diff` で確認し、意図外の変更がないか検証すること。
